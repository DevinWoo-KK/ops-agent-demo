反洗钱（AML）

洗钱分三个阶段：放置（现金进入银行）→ 分层（多账户流转隐匿来源）→ 整合（回流为合法资金）。银行的 AML 系统每天产生几千条告警，99% 是误报。

Agent 不是替代规则引擎，而是接在引擎后面——告警来了，自动调取关联交易、图谱分析资金链路、核查制裁名单、检索历史案例，给出可解释的判断。合规人员只复核标为"可疑"的案例。

为什么选择 ReAct Agent 而非纯规则引擎

规则引擎只能处理已知模式，面对新型洗钱手法（如 NFT 洗钱、跨链混币）束手无策。LLM Agent 具备开放域推理能力，能根据语义理解发现规则未覆盖的可疑模式，同时保留规则引擎作为初筛，避免 Agent 资源浪费。

规则引擎处理 80% 已知告警（零 LLM 成本） → Agent 深度调查剩下 20% 的复杂/新型案例。

8 个调查工具Tools

任务规划器（Planner）

接收初始任务（如「账户 ACC-88812 存在可疑交易，请调查」），拆解为子任务序列并通过 System Prompt 注入 FATF 40 条建议等 AML 专业框架，引导 LLM 按合规逻辑推理。

示例拆解: ①查询近30天交易流水 ②与历史基线对比识别异常 ③图谱分析追踪资金流向 ④比对黑名单 ⑤检索相似历史案例 ⑥综合评估风险 ⑦生成 SAR 草稿

工具调用引擎（Tool Router）

根据 Agent 的 Action 决策路由到对应 Tool，管理并发调用、结果聚合和异常处理。支持 ReAct 循环中的多轮工具调用。

记忆与上下文管理

短期记忆当前调查 Session 内的所有工具调用结果，维护在 Agent 上下文窗口中
长期记忆历史案件结论存入向量数据库，支持 case_retrieval 检索
状态持久化每次工具调用结果落库，支持断点续跑
多轮对话审查员可通过自然语言追问，Agent 基于已有上下文继续深挖

8 个调查工具详细规格

transaction_query
查询指定账户在时间范围内的交易记录，返回列表+统计摘要（总笔数/总金额/夜间占比等）
输入: account_id, start_date, end_date, filters
graph_analysis
资金流向图中发现关联账户、循环路径、团伙结构。N跳路径+社区发现(Louvain)
输入: account_id, depth, analysis_type
rule_engine
逐条检查预设 AML 规则，输出命中列表+依据+置信度+监管条款。内置规则如分散转入、制裁关联等
输入: account_id, rule_set
risk_scorer
综合评分 0-100，分项: 行为异常度/关联风险/黑名单匹配/历史违规。支持 XGBoost 或 LLM 评分
输入: account_id, context
sanction_check
核查 OFAC/UN/央行黑名单。精确匹配+向量模糊匹配（防拼写变体逃避）
输入: name, id_number 或 account_id
case_retrieval
向量语义检索 Top-K 相似历史 AML 案例，含结论/处罚结果
输入: 案件摘要文本
news_search
检索客户/企业相关负面新闻，标注情感倾向和风险关键词
输入: entity_name
report_generator
基于调查结论生成符合监管格式的 SAR 报告草稿(Markdown/PDF)
输入: 调查摘要/风险评级/命中规则/证据

附录：关键设计决策说明

为什么要保留人工审核？

AML 决策（尤其是账户冻结和 SAR 上报）具有法律效力，AI 误判代价极高。Human-in-the-Loop 不仅是合规要求，也是建立系统公信力、收集标注数据持续优化的必要环节。

如何控制 Agent 的幻觉风险？

所有结论必须有工具调用的实际数据支撑，禁止 Agent 在没有工具返回结果的情况下编造数据。通过 Prompt 约束要求 Agent 明确区分「已证实事实」与「推断/假设」，并在 SAR 报告中分别标注。
