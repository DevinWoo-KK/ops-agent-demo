什么是金融IT运维

每次用手机银行转账，背后是 6 道工序、200 毫秒内完成、每天几千万次。任一环节故障 = 交易失败。

手机App → 前置服务器 → 交易系统 → 数据库 → 存储落盘 → 对端清算

金融IT运维就是保证这条链永远不断。磁盘满了、连接池耗尽了、存储I/O延迟飙了——任一环节故障，就是真金白银的损失和监管问责。

为什么需要 AI Agent

现在的痛点

· 告警风暴：每天几百条，99%是误报
· 依赖专家：凌晨告警等人工响应
· 处理不均：水平参差，漏判误判
· 重复劳动：80%告警处理套路固定

Agent 化的价值

· MTTD 从几十分钟→几秒
· MTTR 从几小时→几分钟
· 低级故障全自动处理
· 人只处理真正复杂的场景

📡 感知  Monitor
实时采集鲲鹏、OceanStor、GaussDB 等多层设备指标。连续3-18次采样确认，排除传感器抖动和瞬时波动。
🧠 分析  Analyze + LLM
LLM 多维数据推理，输出根因判断 + 3-5条证据链 + 影响评估 + 推荐操作。不是"磁盘满了就清理"。
⚖️ 决策  Decide
Agent 自主判断：可自动执行 / 需人工确认 / 必须人工决策。三级分级，每级有明确的安全边界。
🔧 执行  Remediate
自动执行修复操作：归档日志、调整配置、KILL 异常连接、通知业务方。每步记录结果，可回滚。
📋 报告  Report
生成结构化事件报告：设备信息 + 告警详情 + 根因分析 + 修复记录 + 后续建议。满足金融监管审计要求。

Skill 声明式体系

每个Skill独立定义：触发条件、上下文依赖、工具链、输出规范、失败降级。声明式优先——让非技术人员也能理解和维护 Agent 行为。

monitor
触发: 定时采集/告警推送
依赖: 无
降级: 人工检查监控面板
analyze
触发: 异常检测
依赖: monitor
降级: 转人工专家分析
remediate
触发: 可自动修复
依赖: analyze
降级: 生成工单指派工程师
report
触发: 所有Skill完成
依赖: remediate
降级: 发送原始告警邮件

自主决策的三级分级

全自动执行

无风险操作——日志归档、配置调整、流量降级。Agent自主判断+执行。

部分人工确认

中等风险——KILL数据库连接、调整连接池。Agent执行后需人工通知业务方。

必须人工决策

高风险——RAID重建中断、灾备切换、涉及资金的操作。Agent只做分析和建议。

Demo → 产品：差距与方案

Demo 证明了架构可行。以下是推向生产环境需要解决的 6 个核心差距及解决方案。

① 数据层：模拟 vs 真实监控

差距
Demo 数据是手写的 3 个场景。真实环境每天数百条告警，来自 Prometheus、Zabbix、华为 iMaster NCE 等多个系统，需要告警去重、聚合、定级。

方案
对接标准告警接口（Prometheus AlertManager API / 华为 iMaster 北向接口），规则引擎做第一层去重收敛，异常事件才进入 Agent 分析。

② 安全围栏：Agent 能做什么不能做什么

差距
Demo 中 Agent 可以直接 KILL 数据库连接、修改服务器配置。金融行业要求操作分级审批 + 堡垒机审计，不能有"自动化后门"。

方案
三级操作白名单：L1 只读操作直接执行，L2 配置变更需审批，L3 高危操作必须人工在堡垒机上执行。每步操作记录完整审计日志。

③ 多租户：同时管多家银行

差距
华峻捷科的客户（省农信社、广州农商行、中邮消费金融...）数据必须绝对隔离。同一套 Agent 不能串数据。

方案
Workspace 级隔离：每个客户独立配置空间、独立 Skill 规则集、独立 LLM 上下文。Agent 实例按客户隔离部署或逻辑分区。

④ 学习闭环：这次修了下次能复用吗

差距
Demo 的分析是预写好的。真实 Agent 需要从每次处理中积累知识——这次遇到 RAID 重建不知道怎么处理，人工处理完后 Agent 应该学会。

方案
事件知识图谱：每次处理的事件→根因→修复方案存入图数据库。下次相似告警直接匹配历史方案。SSOT（Single Source of Truth）持续更新。

⑤ 成本控制：每次 LLM 分析花多少钱

差距
Demo 不限成本调 LLM。真实场景每天数百条告警，全量 LLM 分析成本不可接受。华为 ModelArts 上推理也有算力成本。

方案
三层过滤：规则引擎处理 80% 已知告警（零 LLM 成本）→ 轻量模型做二级判断 → 仅复杂/新型告警调大模型。历史匹配命中则直接复用结果。

⑥ Agent 自身的运维：谁来监控监控者

差距
Demo 假设 Agent 永远正常运行。真实环境 Agent 本身也会挂、API 会超时、模型响应会变慢。银行不能接受"Agent 挂了所以告警没人处理"。

方案
Agent 双活部署 + 心跳监控。超过 N 秒未响应则自动切换备节点。同时保留传统告警通道（短信/电话）作为最后兜底，Agent 不是唯一路径。
